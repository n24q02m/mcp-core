## 2025-02-28 - [CRITICAL] Fix command injection in browser relay
**Vulnerability:** Command injection was possible in `try_open_browser` inside WSL because `cmd.exe /c start` was used to open URLs with inadequate escaping (only `&` was escaped). An attacker could pass a malicious URL like `http://example.com | calc.exe` to execute arbitrary commands.
**Learning:** `cmd.exe /c start` should be avoided when processing untrusted input due to shell evaluation. It is more secure to use an executable directly, like `explorer.exe`, or properly structure arguments without going through a shell. Additionally, strictly validating URLs (e.g., verifying `http/https` scheme and stripping shell metacharacters like `|`, `<`, `>`, `"`, `'`, \`, `\`) adds a layer of defense-in-depth. However, it's crucial to remember that `&` is a valid character in legitimate URLs.
**Prevention:** Always sanitize arguments passed to subprocesses and avoid invoking processes through a shell (`cmd.exe /c`) whenever possible. For browser opening, use secure executables like `explorer.exe` on Windows/WSL and validate the input against allowed schemes and potentially dangerous characters.

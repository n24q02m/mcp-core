## 2025-04-12 - Cached PBKDF2 Key Derivation
**Learning:** The config file encryption used PBKDF2 with 600,000 iterations inside `deriveFileKey()`, which was being called on *every* config file read and write. This caused severe performance bottlenecks, taking over 3 seconds to execute a simple loop of 10 reads. Caching the derived key in memory significantly reduces this CPU overhead (down to ~5ms for the same loop).
**Action:** When working with cryptography in this codebase, specifically PBKDF2 key derivations in `packages/core-ts`, ensure the result is cached for subsequent operations within the same process lifecycle to avoid unnecessary CPU blocking. Remember to provide a way to clear the cache for isolated unit testing (e.g., `clearKeyCacheForTesting`).
## 2025-04-14 - [Cache PBKDF2 iterations in configuration reads]
**Learning:** High PBKDF2 iteration counts (e.g., 600,000 for `mcp_core.storage.config_file` decryption key derivation) cause significant CPU delays (~350ms) per call. The TypeScript codebase cached this internally to optimize sequential operations, but the Python implementation did not have parity, leading to unexpected latency during repetitive config interactions (like reading/writing configurations or setting up servers).
**Action:** When working on cryptographic workflows or key derivation, ensure that keys generated via expensive operations (like PBKDF2, Argon2) are cached per-process if they are deterministic for the session (e.g. derived from static machine ID/username). Always check for cross-language parity when porting cryptographic wrappers.
